# CMP 云管理平台 — 权限体系说明

> 整理自原型，说明平台超管、部门负责人、一级组组长、二级组组长、普通用户各自能做什么操作。

---

## 一、权限体系概述

CMP 存在**两套独立的权限体系**，互不干扰：

| 权限类型 | 说明 | 管理方 |
|------|------|------|
| **功能模块权限**（角色权限） | 控制用户能访问哪些页面、能做哪些操作（如创建部门、审批申请等） | 超管创建角色；部门负责人为本部门成员分配角色 |
| **资源权限**（reporter / developer / master） | 控制用户对某个具体云资源的操作级别 | 通过资源包或直接授权 |

本文聚焦**组织身份**维度的权限：平台超管、部门负责人、一级组组长、二级组组长、普通用户。这五个身份由**组织架构**决定（不是角色管理里的"角色"），系统按身份自动赋予对应权限。

---

## 二、各身份权限详解

### 2.1 平台超级管理员

**是什么：** 平台级内置角色，拥有所有功能模块的全部权限，不受部门限制。

#### 组织管理
- 创建 / 编辑 / 删除**部门**（仅超管可操作，组长和部门负责人无此权限）
- 管理**未分配成员**：查看未归属任何部门的成员，并将其分配到指定部门
- 创建 / 编辑 / 删除**角色**，配置角色的功能模块权限点

#### 用户管理
- 手动**创建用户**（正常用户由 ERP 自动同步，超管可创建特殊账号）
- 查看所有用户，为任意用户分配角色

#### 资源配置（平台级）
- 管理**资源目录**：定义平台支持的云资源类型、支持的操作（创建/变配/扩容/销毁等）
- 配置**平台级资源申请模板**：定义每类资源申请时的表单字段和云 API 对接参数
- 配置**平台级审批流模板**：定义每类资源申请的标准审批链路

#### 操作审计
- 查看全平台所有用户的操作日志（跨部门）

#### 资源申请审批
- 可被配置为资源申请流中的**指定审批人**（admin1 / admin2 节点），对特定资源类型的申请进行最终审批

---

### 2.2 部门负责人

**是什么：** 在组织架构中被指定为某部门负责人的成员。每个部门有且仅有一个部门负责人。

#### 组织管理（本部门范围）
- 创建 / 编辑 / 删除本部门下的**一级组和二级组**
- 指定或变更组长
- 将成员**归组**：把停留在部门直属层的成员分配到某个一级组
- 将成员**移出部门**：成员变为未分配状态，需超管重新分配到部门
- 为本部门成员**分配功能角色**（只能分配已有角色，不能创建角色）

#### 云账号管理（本部门范围）
- **关联 / 更换**本部门的阿里云主账号（每部门限绑一个主账号）
- 查看本部门所有子账号

#### 部门配置（本部门范围）
- **资源申请模板**：在平台模板基础上进行简化定制（如限定可选地域、规格等）
- **资源审批流配置**：为每类资源申请指定 admin1 / admin2 审批人
- **工单处理人配置**：为每个工单类别指定本部门的处理人

#### 资源申请审批
- 作为审批流中的 **"部门负责人（l5）"节点**，审批本部门所有成员提交的资源申请
- 当申请人直属领导与部门负责人为同一人时，合并为一个审批节点

#### 孤儿资源
- 对通过子账号在云上直接创建（绕过 CMP 申请流程）的资源，部门负责人可将其**分配到项目**

---

### 2.3 一级组组长

**是什么：** 在组织架构中被指定为某一级组（group）组长的成员。

#### 组织管理（本组范围）
- 添加成员到本组（从本部门直属成员中选择）
- 将本组成员**调组**：转移到同部门的其他组
- 将本组成员**移出组**：成员回到部门直属层，等待部门负责人重新归组
- 为本组成员分配**资源权限**（reporter / developer / master）

#### 资源申请审批
- 作为审批流中的**"直属领导（leader）"节点**，审批本组成员提交的资源申请
- 常见审批链路：
  - 简单申请：`申请人 → 直属领导（组长） → 部门负责人`
  - 复杂申请：`申请人 → 直属领导（组长） → 部门负责人 → 指定审批人`

> **注意：** 一级组组长**不能**创建或删除子组，子组的创建由部门负责人操作。

---

### 2.4 二级组组长

**是什么：** 在组织架构中被指定为某二级组（subgroup）组长的成员。二级组隶属于一级组。

#### 组织管理（本组范围）
- 添加���员到本二级组
- 将本组成员**调组**（转移到同部门其他组）
- 将本组成员**移出组**（回到一级组或部门直属层）

#### 资源申请审批
- 作为审批流中的**"直属领导（leader）"节点**，审批本二级组成员的资源申请
- 审批链路与一级组组长相同

> 二级组是当前系统支持的最深层级，二级组组长与一级组组长的权限基本对等，区别仅在于管辖范围（二级组 vs 一级组）。

---

### 2.5 普通用户

**是什么：** 已归属某部门/组、未担任任何负责人或组长职位的普通成员。这是人数最多的身份，也是 CMP 的主要使用者。

#### 资源申请
- 在本部门支持的资源类型范围内，通过 CMP 表单**发起资源申请**（创建、变配、扩容、销毁等）
- 查看**自己提交的申请记录**（申请状态、审批进度、表单详情）
- 在申请通过前可**撤回**申请

#### 云账号（子账号）
- 申请归属于自己���**云子账号**（需审批，关联本部门主账号）
- 查看**自己名下的子账号**（子账号仅本人可见）

#### 工单
- 创建工单（账号权限类、资源问题类、网络问题类、安全合规类等）
- 查看**自己提交的工单**及处理进度
- 在工单解决后**确认关闭**

#### 资源
- 查看被授权的资源详情（基于资源权限 reporter / developer / master）
- 对有 developer / master 权限的资源，通过 CMP 发起对应操作申请
- 创建和管理**资源包**（将自己有权限的资源打包，方便分享给团队）

> **注意：** 普通用户**不能**直接操作云资源，��有变更都需通过申请流程，审批通过后由平台自动执行。

## 三、审批流说明

资源申请走的是逐级审批流程，链路由部门配置决定：

| 流程模板 | 节点顺序 |
|------|------|
| `leader` | 申请人 → 直属领导 |
| `leader + l5` | 申请人 → 直属领导 → 部门负责人 |
| `leader + l5 + admin1` | 申请人 → 直属领导 → 部门负责人 → 指定审批人 |

**直属领导（leader）** = 申请人所在组的组长（一级组或二级组组长）

**部门负责人（l5）** = 申请人所在部门的部门负责人

**指定审批人（admin1/admin2）** = 由超管或部门负责人在审批流配置中指定，通常为超管或运维负责人

## 五、补充说明

### 关于"角色"与"组织身份"的区别

- **组织身份**（部门负责人 / 组长）由组织架构位置决定，系统自动识别，主要影响**审批链路**和**本组织范围内的管理操作**。
- **功能角色**（超级管理员 / 运维负责人 / 审核人 / 工单处理人）由超管或部门负责人显式分配，控制**页面和功能的访问权限**。
- 两者叠加生效。例如，一位部门负责人同时被分配了"运维负责人"角色，则同时具备两套权限。

### 关于资源权限（reporter / developer / master）

资源权限独立于上述两套权限，通过**资源包**统一授权或单独授权给用户：

| 级别 | 含义 |
|------|------|
| reporter | 只读，可查看资源信息 |
| developer | 可操作资源（变配、扩容等） |
| master | 完全控制，包括销毁 |
